# Repository Guidelines

## Project Structure & Module Organization

This is a full-stack MNIST classroom competition app. The React/Vite frontend lives in `src/`: pages are in `src/pages/`, shared UI in `src/components/`, browser helpers in `src/lib/`, hooks in `src/hooks/`, styles in `src/styles/`, and frontend tests in `src/test/`. The FastAPI backend lives in `backend/`, with the app implementation under `backend/app/`, tests in `backend/tests/`, and local runtime data under `backend/data/`. Root `data/` contains frontend sample assets, `scripts/` contains local start helpers, and `dist/` is generated by the frontend build.

## Build, Test, and Development Commands

- `npm install`: install frontend dependencies.
- `python -m pip install -r backend/requirements.txt`: install backend dependencies.
- `npm run dev`: start the Vite frontend at the default local port.
- `uvicorn backend.main:app --reload`: start the backend locally.
- `npm run start:windows` or `npm run start:linux`: start the project with the provided platform script.
- `npm run build`: produce the frontend production build in `dist/`.
- `npm run preview`: preview the built frontend.
- `npm run test:frontend`: run Vitest tests.
- `npm run test:backend`: run Python `unittest` tests.
- `npm run test:stage2`: run backend and frontend tests together.

## Coding Style & Naming Conventions

Follow `.editorconfig`: UTF-8, LF endings, final newline, spaces, 2-space indentation by default, and 4-space indentation for Python. Use React component files in PascalCase, such as `BeginPage.jsx` and `AppChrome.jsx`. Keep page-specific CSS in `src/styles/` with clear feature names, such as `training.css`. Backend tests use `test_*.py`; frontend tests use `*.test.jsx`.

## Testing Guidelines

Add or update targeted tests when changing behavior. Place frontend tests in `src/test/` and backend tests in `backend/tests/`. Prefer `npm run test:stage2` before handing off cross-stack changes. Use `npm run test:stage3:soak:quick` for short stability checks and `npm run test:stage3:soak` for longer load verification.

## Commit & Pull Request Guidelines

Recent history uses short, direct commit subjects such as `Make deployment config nginx-friendly` and release labels like `v1.0`. Keep subjects concise and outcome-focused. Pull requests should describe the user-visible change, list test commands run, call out environment or database effects, and include screenshots for UI changes.

## Security & Configuration Tips

Use `backend/.env.example` as the configuration reference. Do not commit local SQLite databases, generated admin tokens, or private environment values. Configure `ADMIN_TOKEN`, `CORS_ORIGINS`, `ADMIN_UI_BASE_URL`, and `VITE_API_BASE_URL` explicitly for deployed environments.
